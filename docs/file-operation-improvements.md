# File Operation Improvements Plan

Prism FM のコピー・移動・削除操作の信頼性と処理速度を改善するための計画。

## 現状の課題

現在の `batchFileOperation` (main.js) は以下の問題を抱えている:

- `fs.copyFile` による一括コピーのため、大きいファイルで進捗が停止して見える
- 1ファイルのエラーで操作全体が失敗する
- パーミッション・タイムスタンプが保持されない
- ディスク容量不足時にコピー途中で失敗する
- クロスデバイス移動でコピー成功・削除失敗のリスクがある
- 全ファイルが逐次処理で、小さいファイルの大量コピーが遅い
- 大量ファイルの削除で進捗表示がない
- シンボリックリンクがリンク先の実体としてコピーされる

## 改善項目

### 1. ストリームコピーとチャンク進捗 (速度・UX)

**対象**: `copyRecursive` 内の `fs.copyFile` 呼び出し

**変更内容**:
- `fs.createReadStream` / `fs.createWriteStream` によるストリームコピーに置換
- 64KB チャンクごとに `processedBytes` を更新し、`reportProgress()` を呼び出す
- 大きいファイル (例: 100MB以上) でストリームコピーを使用、小さいファイルは `fs.copyFile` のまま (オーバーヘッド回避)

**効果**: 数GBファイルのコピー中もプログレスバーが滑らかに動く

### 2. 個別エラースキップ (信頼性)

**対象**: `batchFileOperation` のメインループ

**変更内容**:
- 各アイテムの処理を try-catch で囲む
- エラー発生時、エラーリストに追加して次のアイテムに進む
- 戻り値に `{ success: true, errors: [...], skipped: N, completed: N }` を返す
- レンダラー側で結果通知に反映 (例: "15 items copied, 2 failed")

**効果**: 1ファイルの権限エラー等で操作全体が中断されない

### 3. パーミッション・タイムスタンプ保持 (信頼性)

**対象**: `copyRecursive` のファイルコピー後

**変更内容**:
- コピー元の `fs.stat` で `mode`, `mtime`, `atime` を取得 (scan 時に既に stat している)
- コピー後に `fs.chmod(dest, stats.mode)` と `fs.utimes(dest, stats.atime, stats.mtime)` を実行
- ディレクトリも同様にメタデータを保持
- Windows では `fs.chmod` はno-op なので影響なし

**効果**: コピー先のファイルが元のパーミッションとタイムスタンプを維持

### 4. ディスク容量事前チェック (信頼性)

**対象**: `batchFileOperation` の scan 完了後、コピー開始前

**変更内容**:
- コピー先パーティションの空き容量を `getDiskSpace()` (既存関数) で取得
- `totalBytes` と比較し、不足している場合はエラーを返す
- 移動操作で同一デバイスの場合はチェックをスキップ (rename はデータコピーしないため)

**効果**: 容量不足による中途半端なコピーを事前防止

### 5. 移動のコピー検証 (信頼性)

**対象**: `batchFileOperation` の move 処理 (rename 失敗時のフォールバック)

**変更内容**:
- コピー完了後、コピー先のファイルサイズを `fs.stat` で検証
- 元ファイルとサイズが一致した場合のみ `fs.rm` で元ファイルを削除
- 不一致の場合はコピー先を削除してエラーとする

**効果**: コピー失敗時にソースファイルが消失するリスクを排除

### 6. 小ファイル並列処理 (速度)

**対象**: `copyRecursive` のディレクトリ処理

**変更内容**:
- ファイルサイズの閾値を設定 (例: 1MB)
- 閾値以下のファイルは `Promise.all` で並列コピー (同時実行数: 4〜8)
- 閾値以上のファイルは逐次処理 (I/O帯域を占有しないよう)
- `p-limit` 等の外部依存なしで、シンプルなセマフォを実装

**効果**: 小さいファイルが数千個あるディレクトリのコピー速度が大幅に向上

### 7. 削除の進捗表示 (UX・信頼性)

**対象**: 削除操作全般 (`deleteItem`, `trashItem`)

**変更内容**:
- 削除前にファイル数をカウント
- 再帰的に削除しながらカウンタを更新、`reportProgress()` でUIに反映
- `fs.rm({ recursive: true })` の代わりに自前の再帰削除を使用

**効果**: 大量ファイルの削除で進捗が見える

### 8. シンボリックリンク対応 (信頼性)

**対象**: `copyRecursive` と `scan`

**変更内容**:
- `fs.stat` を `fs.lstat` に変更してリンク自体の情報を取得
- シンボリックリンクの場合、`fs.readlink` でリンク先を取得し、`fs.symlink` で再作成
- scan でリンク先を辿らないため、循環参照による無限ループも防止

**効果**: シンボリックリンクがリンクとして正しくコピーされる

## 実装順序

信頼性に直結するものを優先:

| Phase | 項目 | 理由 |
|:------|:-----|:-----|
| Phase 1 | #2 個別エラースキップ | 最も基本的な信頼性改善 |
| Phase 1 | #5 移動のコピー検証 | データ損失リスクの排除 |
| Phase 1 | #8 シンボリックリンク対応 | コピーの正確性 |
| Phase 2 | #3 パーミッション保持 | メタデータの正確性 |
| Phase 2 | #4 ディスク容量チェック | 事前エラー防止 |
| Phase 3 | #1 ストリームコピー | 大ファイルのUX改善 |
| Phase 3 | #6 小ファイル並列処理 | 速度改善 |
| Phase 3 | #7 削除の進捗表示 | UX改善 |

## 影響範囲

- `main.js`: `batchFileOperation`, `copyRecursive`, `copy-item`, `move-item`, `deleteItem`
- `renderer/views-actions.js`: 結果通知の表示変更
- `renderer/core.js`: 進捗表示の更新
- `preload.js`: 新規IPCが必要な場合のみ

## 互換性

- 既存のキャンセル機能 (`cancelOperations`) はそのまま維持
- 既存の競合ダイアログ (`resolveConflict`) はそのまま維持
- 操作キュー (`enqueueOperation`) のインターフェースは変更なし
- undo スタックへの影響なし
